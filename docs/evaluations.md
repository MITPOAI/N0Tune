# Evaluations

How we measure whether N0Tune is doing what it claims.

## Running the suite

```bash
python -m pip install -e "apps/api[dev]"
python -m evals                       # run every registered eval
python -m evals token_savings         # run a specific eval
```

Evals boot an in-process FastAPI app against an isolated in-memory SQLite
engine via [`evals/harness.py`](../evals/harness.py). No Docker, no real
provider, no network — the suite is meant to be fast enough to run in CI on
every PR.

## What's in the directory

```
evals/
├── harness.py                    # fresh_test_client + emit_report
├── __main__.py                   # `python -m evals` registry runner
├── token_savings_eval/           # IMPLEMENTED
│   ├── scenarios.json
│   └── run.py
├── memory_relevance_eval/        # placeholder — needs labelled dataset
├── context_compression_eval/     # placeholder — see token_savings for the pattern
├── prompt_injection_eval/        # placeholder — detection logic tested in test_hardening
├── semantic_cache_eval/          # placeholder — invalidation tested in test_hardening
└── answer_quality_eval/          # placeholder — needs LLM judge or human labels
```

Placeholders ship as README-only directories so future PRs can land their
implementations without re-creating the convention.

## token_savings_eval

The one shipped eval. It seeds a realistic scenario into an isolated API,
asks every query, and compares the compiled prompt against a hand-rolled
"stuff everything in" baseline.

### Methodology

1. Seed memories. The scenario has 23 memories across two users — about half
   are RAG-related, the rest are deliberate noise (coffee preferences,
   vacation, laptop choice, MBA studies).
2. Seed documents. The scenario has 12 short documents, 5 of them about RAG
   and 7 about company internals that have nothing to do with the question.
3. For each query, call `POST /v1/context/preview` and record
   `prompt_tokens_estimated`.
4. Compute a `naive_prompt_tokens` baseline locally:
   - Verbose system prompt (78 tokens-ish)
   - 8 turns of fake chat history
   - **Every** memory belonging to the user
   - **Every** document, in full
   - The current question
5. Report `tokens_saved = naive - compiled` and `tokens_saved_percent`.

The naive baseline is **not** generated by the API — it's the eval's honest
simulation of the most common shape developers actually ship: long system
prompt, full history, stuffed memories, stuffed corpus.

### Why the percent is the headline number

Absolute token counts depend on prompt verbosity, history length, and corpus
size. The percentage saved is the cleaner signal because it scales: a real
production app with 50 memories per user and a 500-chunk corpus will see
much bigger absolute savings while the percent stays in the same band.

### Limitations

- **Hash embeddings.** Tests use the deterministic `hash` backend so they
  stay keyless. Real embeddings (OpenAI / fastembed) would select better,
  which would raise savings further. The eval underestimates rather than
  overstates.
- **Single scenario.** One scenario, two users, twelve queries-worth of
  retrieval. Add more scenarios as we accumulate them.
- **Naive baseline is a guess.** Some teams already do something smarter
  than "shove everything in." The baseline reflects what we see in the
  wild, not the worst possible alternative.

## Adding a new eval

1. Create `evals/<name>_eval/` with `__init__.py` and `run.py`.
2. Expose a `run() -> dict[str, Any]` function. Use `fresh_test_client()`
   from the harness; clean up between scenarios.
3. Register the eval in `evals/__main__.py`'s `REGISTRY`.
4. Add `scenarios.json` (or whatever input format fits) alongside `run.py`.
5. Write a README explaining the methodology and limitations.
6. Add the eval's headline number to [docs/benchmarks.md](benchmarks.md).

## What this is not

- Not a leaderboard. The point is to make N0Tune's claims falsifiable, not
  to compare against other products.
- Not a unit test. The suite is allowed to be slower and dependent on real
  scoring behavior; unit tests live in `apps/api/app/tests/`.
- Not a substitute for production telemetry. Once N0Tune is deployed,
  [Langfuse traces](observability.md) and the `context_runs` table give you
  real-user numbers; evals just keep the lower bound honest.
