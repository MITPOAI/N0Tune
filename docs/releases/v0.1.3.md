# N0Tune v0.1.3

> The body to paste into the GitHub release for v0.1.3.

A "make it smart and wire it" release. v0.1.2 made N0Tune downloadable;
v0.1.3 makes it **actually work** in a live Claude Code session and adds
a real alignment layer so it isn't just context amplification.

## Smarter retrieval (the "layer" the user kept asking about)

The context compiler now stacks three signals on top of vector + lexical
similarity, all computed in-process with no LLM calls:

- **State weighting.** `CONFIRMED` memories (explicitly affirmed or
  written by the consolidation pass) get a 1.10× multiplier, `ACTIVE`
  stay at 1.0×, `CANDIDATE` get 0.80×. Clamped to [0, 1].
  ([apps/api/app/services/memory/lifecycle.py](apps/api/app/services/memory/lifecycle.py))
- **Type-aware decay half-life.** Preferences (stable traits) decay
  over 180 days, facts over 90 days, project state over 30 days.
  Previously every memory used the same 60-day half-life regardless of
  type.
- **MMR diversity.** After the vector+lexical ranking, a greedy
  max-marginal-relevance pass drops near-duplicates (cosine ≥ 0.92).
  Stops the compiler from spending its token budget on five paraphrases
  of the same fact. Dropped memories are surfaced in the context trace
  as `near-duplicate of mem_<id>`.
  ([apps/api/app/services/context/compiler.py](apps/api/app/services/context/compiler.py))

Live-stack smoke: 3 duplicate memories about the same fact → 1
selected, 2 dropped by MMR with full trace citing the canonical memory.

## Context Guard CG-1 + CG-2 + CG-5 minimal — the alignment layer

CG-0 shipped as docs in v0.1.2. v0.1.3 makes it **real**:

- **CG-1 — rule engine.** Deterministic, zero LLM, zero network
  egress. Six rule types plus an always-on secret detector:
  `terminology`, `phase_scope`, `forbidden_claim`, `required_test`,
  `security`, `memory_policy`. 18 unit tests cover the engine.
  ([apps/api/app/services/alignment/rules.py](apps/api/app/services/alignment/rules.py))
- **CG-1 — alignment_rules table** + alembic migration (`20260519_0004`).
- **CG-2 — REST API.** `POST /v1/alignment/check` (run the engine),
  `GET /v1/alignment/rules` (public — list active rules so agents
  know what they're checked against), `POST /v1/alignment/rules`
  (admin-only, RBAC: owner). 5 endpoint integration tests.
  ([apps/api/app/routes/alignment.py](apps/api/app/routes/alignment.py))
- **CG-5 — MCP tool.** `n0tune_alignment_check` is now the eighth tool
  on the MCP server. Agents call it to ground a plan before committing
  claims to the user. Smoke-verified through JSON-RPC end-to-end.
- **Starter rule set.** `scripts/seed-alignment-rules.py` seeds seven
  rules (terminology × 2, security, benchmark inflation, required
  tests, phase-scope, memory policy) idempotently. The script is the
  blueprint for the rules a fresh installation should start with.

## Wiring fix for Claude Code in worktrees (now actually works)

v0.1.2 introduced `n0tune mcp sync` + `npm install` postinstall. v0.1.3
fixes a real bug discovered while smoke-testing: Claude-Code-managed
worktrees under `.claude/worktrees/` are a **sparse copy** of the
project — they don't have `integrations/`. The relative path
`./integrations/mcp-server/src/server.mjs` from the canonical config
therefore fails when Claude Code spawns the subprocess from a worktree.

The sync script now **rewrites relative args to absolute paths** when
the target worktree is a sparse copy that doesn't contain the
referenced files. The canonical config stays unchanged; only the
worktree copies get rewritten. Verified end-to-end: spawning the MCP
server from a worktree's cwd now succeeds, lists 8 tools, and serves
real JSON-RPC traffic.

[`docs/wire-to-claude.md`](../wire-to-claude.md) now also includes a
one-screen restart procedure walking through:
1. Boot the Gateway (`docker compose up`).
2. Sync the MCP config (`n0tune mcp sync` or `npm install`).
3. Restart Claude Code from any synced directory.
4. Verify with `/mcp` listing — and a smoke-test prompt that drives
   `n0tune_search_memories` from inside Claude.

## Honest framing — what N0Tune actually does

N0Tune is a **context amplifier**, not a code-quality engine. When wired
in, it:

1. Reads your message → embeds it.
2. Retrieves the relevant memories + file chunks for *you*.
3. Applies your persona.
4. Fits everything into a token budget.
5. Drops anything injection-risky.
6. Checks the semantic cache.
7. Sends the compiled prompt to **your** provider.
8. Saves new preferences silently, in the background.
9. Consolidates similar memories over time (auto-trigger from v0.1.2).
10. **CG-1 (new): checks claims against stored rules** — flags
    terminology drift, security risks, secret leaks, benchmark
    inflation, missing tests, phase drift.

Same model. Personal answer. Fewer wasted tokens. Now with active
drift detection.
