# N0Tune v0.1.2

> The body to paste into the GitHub release for v0.1.2.

A framing + dynamics release. The product surface from v0.1.1 stays the
same; what changes is the **headline** (back to context-tuning, not
"armor"), the **continual-learning loop** (now dynamic, not manual), and
the **download path** (a cross-platform release workflow that auto-builds
and attaches installers).

## Headline (corrected)

**N0Tune. Fine-tune any AI. Without fine-tuning.**

v0.1.1 over-pivoted to "armor for your AI tools." That framing made
N0Tune sound like a wrapper / accessory, which understates what the
system actually does. The honest one-line is back to **context-tuning**:
bring any model — OpenAI, Anthropic, Gemini, Qwen, OpenRouter, Ollama,
LM Studio, anything OpenAI-compatible — N0Tune adds memory, persona,
indexed files, semantic cache, and a context compiler that builds a
compact personal prompt per request. Same model, personal answer.

Both consumption surfaces are now framed as first-class:

- **Standalone:** the N0Tune Desktop app (real Tauri build, local
  SQLite + OS keychain). Your personal AI on your machine.
- **Integration layer:** MCP server + OpenAI-compatible proxy + SDKs.
  For Claude Code / Cursor / Codex CLI / ChatGPT-shaped clients / your
  own code.

[product-direction](../product-direction.md), [overview](../overview.md),
[editions](../editions.md), [how-it-works](../how-it-works.md),
[install](../install.md), [CLAUDE.md](../../CLAUDE.md),

## Dynamic memory consolidation

The continual-learning loop is now **dynamic**.

Before: consolidation only ran via `POST /v1/memories/consolidate` or
the `n0tune memory consolidate` CLI. The eval scenarios called it; real
chat did not.

Now: after every successful **chat** (`POST /v1/chat`,
`POST /v1/openai/chat/completions`) and **memory write**
(`POST /v1/memories`), a **background task** checks two cheap conditions
in O(1) DB queries:

1. The user has at least 12 active memories
   (`AUTO_MIN_ACTIVE_MEMORIES`).
2. The most recent consolidation summary for that user is older than 15
   minutes (`AUTO_COOLDOWN`).

If both pass, the full clustering pass runs in a fresh session: it
finds clusters of memories with cosine similarity ≥ 0.85, collapses
each cluster into a single summary, and marks the originals
`state="deprecated"` with `replaced_by_memory_id` pointing at the
summary. The compiler's existing `is_retrievable` filter already
excludes deprecated rows, so future chats see only the denser summary.

End-to-end smoke (verified against the live stack): writing 13 similar
memories triggered consolidation after the 13th write; 2 of them
collapsed into 1 summary; state was 12 active + 1 summary + 2
deprecated, no LLM tokens spent (consolidation uses the deterministic
embedding clustering, **not** the chat provider). The provider-backed
summary path stays opt-in via `summarize=True` on the explicit
endpoint, for users who want richer summaries and are willing to pay
the LLM call.

### Consolidation runs separately from the chat provider

To answer the obvious question — when you use Codex, OpenAI, Claude, or
any provider, does consolidation share that provider?

**Today the auto-trigger does not call any provider.** It's a pure
embedding-similarity clustering pass that runs against the local
embedding backend. Cost: zero tokens. Speed: O(N²) cosine in a single
database round-trip, fired as a background task so it never blocks the
chat response.

If you opt into `summarize=True` on the explicit endpoint, the LLM-backed
summary call uses the **same** provider you configured for chat (via
`N0TUNE_PROVIDER_*` env vars). There is no separate consolidation
provider yet; adding one is a v0.1.3 follow-up if there's demand.

## Cross-platform release workflow

[`.github/workflows/release.yml`](../../.github/workflows/release.yml)
fires on every `v*` tag push. It builds the Tauri bundle on
`windows-latest`, `macos-14` (arm64), `macos-13` (x64), and
`ubuntu-22.04`, collects every `.exe / .msi / .dmg / .AppImage / .deb`
under `apps/desktop/src-tauri/target/<target>/release/bundle/`, and
uploads them to the matching GitHub Release as assets. The release
itself is still created with the body from
`docs/releases/<tag>.md`; the workflow only attaches binaries.

So v0.1.2 is the first release where you can **click and download a
pre-built installer** instead of cloning + building.

Note: binaries are still **unsigned**. Windows SmartScreen and macOS
Gatekeeper will warn on first run; the install doc walks through the
one-time bypass.
